> **Статус:** черновик архитектуры от 2026-06-07. **Код пока не пишем — сначала согласуем.**
>
> SIMBA в чате 2026-06-07 сказал: «откупы должны быть отдельный сервис. логику сохраняем. связь с прайд джарвисом тоже. бот (ЛС + группы) + мини апп как биржа». ## 1. Архитектура верхнего уровня

**Отдельный репозиторий + отдельный Railway-проект.** Имя: `pride-outkup-service`.

Почему отдельный, а не подмодуль workchat-bot:
- Можно делать публичный (доступ извне Telegram-инвайтов) — workchat-bot закрытый, у него много чувствительного.
- Деплой/перезапуски не аффектят основной бот (если упадёт outkup — JARVIS работает).
- Размер кодовой базы workchat-bot уже > 50K строк, добавлять туда биржу — катастрофа.
- Свой набор зависимостей (matching/escrow/KYC библиотеки можно ставить без риска для основного).
- Свой токен бота `@PrideOutkup_bot`, своя WebApp-страница на отдельном Railway-домене.

### Связь с PRIDE JARVIS

JARVIS остаётся **админкой и аналитикой**. Outkup-сервис шлёт ему:

- **Webhook** при каждом событии (новая заявка, выплата отправлена, KYC одобрен, спор открыт). JARVIS пишет в свой storage + показывает в Откупы view.
- **REST API endpoints** (read-only) которые JARVIS может дёргать для refresh:
  - `GET /api/v1/orders?since=ts` — список новых заявок
  - `GET /api/v1/deals?status=active` — активные сделки
  - `GET /api/v1/stats/daily?date=...` — статистика дня для маржи
- **Аутентификация:** HMAC-подпись на webhook'ах + Bearer-token на REST (общий секрет в env).

JARVIS-side: добавится HTTP-клиент `outkup_client.py` который синхронизирует state.outkup_* данные раз в N сек + принимает webhook на endpoint `/api/webhook/outkup`. Старые модули `outkup_detector.py` / Откупы view в JARVIS остаются — но переключаются с локального state на удалённый сервис.

## 2. Стек

| Слой | Технология | Почему |
|------|-----------|--------|
| Telegram bot | aiogram 3.x | Тот же стек что в workchat-bot — знакомо. |
| WebApp backend | FastAPI + uvicorn | Тот же стек. |
| Mini-App frontend | Telegram WebApp SDK + Vue 3 (или vanilla JS если хотим минимализм) | Vue компактнее чем React под P2P-доску. |
| База данных | **PostgreSQL** (через Railway) | Сделки требуют ACID. JSON-state не подходит для concurrent matching. |
| Кэш / live updates | Redis (Railway addon) | Order book live, через polling каждые 2 сек или WebSocket. |
| Платежи | TronGrid + tronpy | Те же библиотеки что уже в `tron_payouts.py` workchat-bot. |
| KYC | Свой простой: КУЦ-видео (как в workchat-bot уже есть) + паспортные данные. Без интеграции SumSub на старте. | |
| Hosting | Railway (отдельный проект) | Тот же провайдер, удобно деплоить. |

## 3. Модели данных

```sql
-- Пользователи биржи
users (
  id BIGSERIAL,
  tg_id BIGINT UNIQUE NOT NULL,
  username TEXT,
  full_name TEXT,
  trc20_address TEXT,
  kyc_status TEXT,  -- 'pending' | 'verified' | 'rejected' | 'banned'
  kyc_data JSONB,   -- паспорт, КУЦ-видео, дата верификации
  invited_by_id BIGINT,
  is_partner BOOLEAN DEFAULT false,  -- наш текущий партнёр или новый клиент
  trust_score INT DEFAULT 0,  -- 0..100, растёт от успешных сделок
  created_at TIMESTAMPTZ,
  updated_at TIMESTAMPTZ
);

-- ╔════════════ V1 МОДЕЛИ — embed-биржа PRIDE ════════════╗
--
-- Order = заявка клиента → PRIDE (мы единственный контрагент).
-- Объединяет 3 типа: «купить USDT за RUB», «продать USDT за RUB»,
-- «откуп бизнес-счёта» (специальный случай купить с большим объёмом).
orders (
  id BIGSERIAL,
  order_number TEXT UNIQUE,     -- '#O0029'
  user_id BIGINT REFERENCES users,
  kind TEXT NOT NULL,           -- 'buy_usdt' | 'sell_usdt' | 'business_outkup'
  amount_rub NUMERIC(12,2),
  amount_rub_remaining NUMERIC(12,2),  -- сколько ещё ждёт реквизита
  rate_rub_per_usdt NUMERIC(10,2),     -- курс PRIDE на момент создания
  amount_usdt NUMERIC(12,2),
  pct_fee NUMERIC(5,2),         -- комиссия % зафиксирована
  destination TEXT,             -- 'balance' (на баланс PRIDE) | 'trc20'
  destination_addr TEXT,        -- если 'trc20'
  bank_in TEXT,                 -- какой банк клиент будет переводить
  status TEXT,                  -- 'pending'|'accepted'|'partial'|'awaiting_receipts'|'done'|'cancelled'
  assigned_to_id BIGINT,        -- кто из откупщиков взял (для kind=business_outkup)
  created_at, updated_at, completed_at
);

-- Issued payment = выданный откупщиком реквизит под Order
-- (для отдельных порций при больших суммах — заявка 50к = 4 порции по 5-15к)
order_payments (
  id BIGSERIAL,
  order_id BIGINT REFERENCES orders,
  payment_number TEXT,          -- 'pay13'
  bank TEXT,
  phone_or_card TEXT,
  amount_rub NUMERIC(12,2),
  manager_username TEXT,        -- кто выдал реквизит (откупщик)
  duration_minutes INT,
  expires_at TIMESTAMPTZ,
  receipt_url TEXT,
  status TEXT,                  -- 'waiting_receipt'|'confirmed'|'rejected'
  created_at, confirmed_at
);

-- ╔════════════ V2 МОДЕЛИ — открытый P2P-стакан ════════════╗
--
-- Offer = объявление пользователя на P2P доске
offers (
  id BIGSERIAL,
  user_id BIGINT REFERENCES users,
  side TEXT NOT NULL,           -- 'buy' (продаёт USDT за RUB) | 'sell' (покупает USDT за RUB)
  rate_rub_per_usdt NUMERIC(10,2),
  min_amount_rub NUMERIC(12,2),
  max_amount_rub NUMERIC(12,2),
  payment_methods TEXT[],       -- ['тинькофф', 'сбер', 'альфа'...]
  conditions TEXT,              -- произвольный текст «работаю с 10:00-22:00»
  status TEXT,                  -- 'active' | 'paused' | 'archived'
  filled_count INT DEFAULT 0,
  cancelled_count INT DEFAULT 0,
  created_at, updated_at
);

-- Deal = сделка между ДВУМЯ юзерами (V2 only)
deals (
  id BIGSERIAL,
  deal_number TEXT UNIQUE,      -- '#D0001'
  offer_id BIGINT REFERENCES offers,
  buyer_id BIGINT REFERENCES users,
  seller_id BIGINT REFERENCES users,
  amount_rub NUMERIC(12,2),
  rate_rub_per_usdt NUMERIC(10,2),
  amount_usdt NUMERIC(12,2),
  payment_method TEXT,
  status TEXT,                  -- 'created'|'awaiting_payment'|'paid'|'released'|'disputed'|'cancelled'
  receipt_url TEXT,
  txid TEXT,                    -- TRC20 transaction hash после release
  expires_at TIMESTAMPTZ,
  paid_at, released_at, cancelled_at
);

-- Escrow-блокировки USDT (для side='sell' откупщиков)
escrow_locks (
  id BIGSERIAL,
  user_id BIGINT,
  amount_usdt NUMERIC,
  deal_id BIGINT,
  status TEXT,                  -- 'locked' | 'released' | 'refunded'
  created_at, released_at
);

-- Споры
disputes (
  id BIGSERIAL,
  deal_id BIGINT,
  opened_by_id BIGINT,
  reason TEXT,
  evidence_urls TEXT[],
  status TEXT,                  -- 'open' | 'investigating' | 'resolved'
  resolution TEXT,              -- 'buyer' | 'seller' | 'split'
  resolved_by_id BIGINT,        -- админ PRIDE
  created_at, resolved_at
);

-- Сообщения внутри сделки (чат между buyer/seller через бота)
deal_messages (
  id BIGSERIAL,
  deal_id BIGINT,
  sender_id BIGINT,
  text TEXT,
  attachment_url TEXT,
  created_at
);
```

## 4. Бот @PrideOutkup_bot — flow

### 4.1. В ЛС бота

- `/start` → приветствие + кнопка «Открыть биржу» (запускает Mini-App)
- `/start ref_USERID` → запоминаем кто пригласил (реферал)
- Если пользователь зарегистрирован и верифицирован → личное меню:
  - 🏛 Открыть биржу (Mini-App)
  - 📋 Мои объявления (offers)
  - 💼 Мои сделки (deals)
  - 💰 Баланс / Вывод
  - 👤 Профиль / KYC
- Если не верифицирован → кнопка «Пройти верификацию» → flow KYC (паспорт + КУЦ-видео + подтверждение TRC20-адреса)

### 4.2. В группах

Бот может быть добавлен в любую группу. Команды:
- `/курс` — текущий лучший курс buy/sell в группе
- `/предложения` — последние N активных offers
- `/сделка @user 50000 рублей` — создаёт сделку между админом группы и упомянутым юзером (только для PRIDE-партнёрских групп — определяется по флагу в users)
- Текст «откуп 100к» (как сейчас) → бот создаёт заявку, ищет оффер с подходящим объёмом, делает матч

### 4.3. Уведомления

Бот шлёт уведомления участникам сделки:
- «Новая сделка #D0001: 50 000₽ → 595.68 USDT. Курс 84.0. Контрагент @user. У вас 15 мин чтобы оплатить.»
- «✅ Контрагент подтвердил получение оплаты. Жди USDT в течение 5 мин на TR...»
- «✅ Сделка #D0001 завершена. TX: tronscan.org/...»
- «⚠️ Контрагент открыл спор. Опиши свою позицию.»

## 5. Mini-App — ДВЕ ВЕРСИИ

**Финальное ТЗ от SIMBA 2026-06-07** (подтверждено вторым уточнением):

> «Мы будем предоставлять функцию откупа рублей в USDT и менять клиентские USDT на рубли. И отдельная вкладка общедоступный P2P — тоже можем начать его разрабатывать. Всё что выше написано в этой вкладке будет.»

То есть:

**V1 (основной Mini-App, мы — единственный контрагент):**
- **Функция A: RUB → USDT** — клиент платит нам рубли через банковский перевод откупщику, получает USDT (на свой баланс PRIDE или сразу на TRC20). Включает обычный обмен и «откуп бизнес-счёта» с большим объёмом и реквизитами по частям.
- **Функция B: USDT → RUB** — клиент даёт нам USDT (списываем с баланса или он шлёт на наш hot-wallet), мы выплачиваем рубли на его банк через откупщика-выплатчика.

Курс единый — наш. Откупщики — наши работники. Маржа = наша.

**V2 (отдельная вкладка «P2P биржа», клиенты ↔ клиенты):**
- Открытая доска объявлений в стиле Binance P2P
- Любой верифицированный юзер создаёт offer со своим курсом
- Другие юзеры выбирают контрагента, делают сделку
- PRIDE — арбитр + escrow + берёт комиссию (например 0.5%) со сделки, не контрагент

Архитектура БД сразу поддерживает ОБЕ модели (см. секцию 3) — но в Mini-App V2 будет отдельной вкладкой, которую открываем когда стабилизируется V1.

**Параллельная разработка возможна** — это два независимых модуля. SIMBA подтвердил что можно начинать разрабатывать P2P параллельно с V1.

## 6. Безопасность и антифрод

### 6.1. KYC

Минимальный уровень для торговли:
1. Паспортные данные (ФИО, серия+номер, прописка) — заполняются в форме
2. КУЦ-видео (как в workchat-bot уже есть): селфи + произнести фразу + показать паспорт
3. TRC20-адрес подтверждается отправкой себе 0.01 USDT обратно (proof of ownership)

Без KYC → можно только смотреть offers, но не создавать сделки.

### 6.2. Escrow

Для side='sell' (откупщик выдаёт USDT): когда создаётся сделка, USDT откупщика **блокируется** в escrow на специальный hot-wallet сервиса. Когда покупатель оплатил RUB и подтвердил, escrow → отпускает USDT на адрес покупателя. Если спор — escrow держит, пока админ не решит.

Это критично — иначе откупщик может «соскочить» после получения RUB.

### 6.3. Антифрод

- Trust score: новые юзеры могут создавать только маленькие сделки (до 10 000₽). Растёт после успешных.
- Rate-limit: не более N сделок в час с одного юзера.
- Blacklist по TG-ID и TRC20-адресам.
- Логирование всех действий с IP (FastAPI).
- Спор-flow: при споре деал заморожен 24 часа, потом админ PRIDE решает.

#### Phase A4: Интеграция с PRIDE JARVIS (0.5 недели)
