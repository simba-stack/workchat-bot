# Карточки ЛК (Группа 1 — «Личные кабинеты»)

## Назначение

Группа 1 — это **доска состояний всех перевязанных ЛК**. Каждый ЛК = одна карточка-анкета. Бот сам создаёт, редактирует и обновляет статусы карточек.

ID группы хранится в настройках (`lk_group_id`). Назначается через админ-роутер.

---

## Формат анкеты

```
1. Поставщик: @username
2. Банк: Альфа
3. ФИО: Иванов И.И.
4. Цена: 400$
5. Метод оплаты: USDT_TRC20
6. Сделка: 12345        (если GUARANTOR_*)
7. USDT-адрес: TXxxx... (если USDT_TRC20)
Статус: В_РАБОТЕ
```

---

## Возможные статусы

| Статус | Что означает |
|---|---|
| `В_РАБОТЕ` | ЛК передан, ждём отработки |
| `ОТРАБОТАН` | Счёт отработан, ЛК готов к выплате/отпуску |
| `ПОПОЛНИТЬ_И_ОТПУСТИТЬ` | По методу GUARANTOR_AFTER_WORK — клиент должен пополнить сделку Continental, после чего отпускаем |
| `БРАК` | ЛК неисправен/невалиден — отдаём обратно поставщику без оплаты |
| `БЛОК` | На счёт пришёл блок (заморозка / арест) — частичная или полная компенсация |
| `ЗАВЕРШЁН` | Деньги выплачены, ЛК закрыт |

---

## Триггеры создания карточки

Карточка создаётся **только когда выполнены оба условия**:

1. **Все данные собраны** (банк, ФИО, цена, метод оплаты, при необходимости — `deal_id`/`usdt_address`).
2. **Перевяз подтверждён** одним из:
   - `@sys01` или `@sys02` подтвердил перевязку в рабочем чате
   - сообщение «🔗 Перевяз ЛК выполнен»

### Если перевяз пришёл раньше данных
Бот **не создаёт карточку**, а:
- запрашивает у клиента недостающие данные
- напоминает каждые **5 минут** (максимум **6 попыток** = 30 мин)
- при поступлении данных проверяет полноту и создаёт карточку

### Если данные собраны раньше перевяза
Бот ждёт перевяз, потом сам создаёт карточку.

## Команды в Группе 1

### `БРАК <card_id> <причина>`
Меняет статус карточки на `БРАК`. Бот:
- уведомляет клиента в рабочем чате
- если был связан `deal_id` — отменяет сделку гаранта

### `БЛОК <card_id> <сумма_₽> <примечание>`
Меняет статус на `БЛОК`. Бот:
- сохраняет сумму блока
- тегает `@TimonSkupCL` для разбора
- при последующем отчёте в Бухгалтерию блок учитывается:
  - если `block_usdt ≥ price` → ЛК **не входит** в маржу
  - если `block_usdt < price` → в маржу идёт **разница** (`price - block_usdt`)

---

## Методы оплаты

| Метод | Что происходит после создания |
|---|---|
| `USDT_TRC20` | Бот сразу подготавливает выплату на указанный адрес после `ОТРАБОТАН` |
| `GUARANTOR_BEFORE_WORK` | Сделка гаранта создана **до** отработки — после `ОТРАБОТАН` сразу отпускаем |
| `GUARANTOR_AFTER_WORK` | Сделка создаётся **после** отработки — бот тегает клиента и просит создать сделку в Conte; после получения номера статус → `ПОПОЛНИТЬ_И_ОТПУСТИТЬ` |
| `CASH` | Наличными — координация через @TimonSkupCL |

---

## Связь с Бухгалтерией

Когда оператор подаёт ЗАЯВКА в Группу 2 со списком ЛК на выходе — бот **автоматически** меняет их статусы на `ОТРАБОТАН`. См. [[accounting_v2]].

## AI-инструмент: `create_lk_card`

Бот может создать карточку через tool-use, если все условия выполнены и есть валидные данные.

Параметры:
- `bank` (обязательно)
- `fio` (обязательно)
- `price_usdt` (обязательно, > 0)
- `payment_method` (обязательно, enum)
- `deal_id` — для `GUARANTOR_*`
- `usdt_address` — для `USDT_TRC20`

Связано: [[accounting_v2]] [[deals]] [[pricing]]
